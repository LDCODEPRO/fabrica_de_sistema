## Foundational Texts

### 1. *Docs Like Code: Collaborate and Automate to Improve Technical Documentation* — Anne Gentle
- **Edition:** 2nd ed., 2019.
- **Publisher:** Lulu Press.
- **Why it matters:** The definitive book on the docs-as-code movement. Covers using Git, pull requests, CI/CD, and developer tools for documentation. Essential for any team integrating documentation into a software development workflow.
- **Key concepts:** Version control for docs, peer review via pull requests, static site generators, automated publishing pipelines.

### 2. *Developing Quality Technical Information: A Handbook for Writers and Editors* — Michelle Carey, Moira McFadden Lanyi, et al.
- **Edition:** 3rd ed., 2014.
- **Publisher:** IBM Press / Pearson.
- **Why it matters:** IBM's internal standards for technical documentation, made public. Defines the nine quality characteristics: easy to use, easy to understand, easy to find, accurate, complete, consistent, concrete, appropriate for the audience, and task-oriented.
- **Key concepts:** Quality characteristics for technical content, structured authoring, minimalism in documentation.

### 3. *The Product is Docs: Writing Technical Documentation in a Product Development Group* — Christopher Gales and the Splunk Documentation Team
- **Publisher:** Splunk Press, 2017.
- **Why it matters:** A rare book written by a documentation team about how to operate within a product organization. Covers the documentation lifecycle, working with engineering, and measuring documentation quality.

### 4. *Every Page is Page One: Topic-Based Writing for Technical Communication and the Web* — Mark Baker
- **Publisher:** XML Press, 2013.
- **Why it matters:** Argues that modern readers land anywhere in documentation via search — they do not read linearly. Every page must therefore be self-contained and navigable in isolation ("every page is page one"). Directly applicable to knowledge base and API documentation design.

## API Documentation

### 5. *Documenting APIs: A Guide for Technical Writers and Engineers* — Tom Johnson
- **Format:** Free online course/guide.
- **URL:** idratherbewriting.com/learnapidoc/
- **Why it matters:** The most comprehensive freely available resource for API documentation. Covers OpenAPI/Swagger, Postman, REST concepts, code samples, and documentation tooling. Used as the primary API documentation reference by DOCS_AGENT.
- **Key concepts:** OpenAPI specification, endpoint reference structure, authentication documentation, code samples in multiple languages.

## Style and Language

### 6. *Google Developer Documentation Style Guide*
- **Format:** Free online guide.
- **URL:** developers.google.com/style
- **Why it matters:** The most widely adopted style guide for developer-facing documentation. Covers voice, tone, formatting, code samples, UI elements, and inclusive language. Reference for all style decisions not covered by project-specific guidelines.

### 7. *Microsoft Writing Style Guide*
- **Format:** Free online guide.
- **URL:** learn.microsoft.com/style-guide
- **Why it matters:** Particularly strong on tone ("warm and relaxed"), accessibility, and inclusive language. Complements the Google guide, especially for product-facing content.

## Standards and Specifications

### 8. *OpenAPI Specification* — OpenAPI Initiative
- **Current version:** 3.1.0 (2021).
- **URL:** spec.openapis.org/oas/v3.1.0
- **Why it matters:** The industry-standard format for describing REST APIs. Machine-readable (YAML/JSON), enabling automated generation of reference docs, interactive consoles (Swagger UI, Redoc), and client SDKs.

### 9. *CommonMark Specification* — John MacFarlane et al.
- **URL:** commonmark.org/spec/
- **Why it matters:** The unambiguous, formally specified version of Markdown. DOCS_AGENT uses CommonMark as the baseline Markdown standard to avoid cross-platform rendering inconsistencies.

### 10. *Semantic Versioning 2.0.0* — Tom Preston-Werner
- **URL:** semver.org
- **Why it matters:** The specification for version numbering (MAJOR.MINOR.PATCH). Essential for structuring changelogs and communicating the nature of releases.

### 11. *Keep a Changelog* — Olivier Lacan
- **URL:** keepachangelog.com
- **Why it matters:** The standard format for human-readable changelogs. Defines sections (Added, Changed, Deprecated, Removed, Fixed, Security) and the principle that changelogs serve humans, not machines.
